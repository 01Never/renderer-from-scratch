# Build System

## The Big Picture

Compiling a C program means turning human-readable source code into a binary the CPU can run. For a single file you could do this directly:

```bash
gcc src/main.c -o renderer.exe -lSDL3
```

But as a project grows — multiple source files, external libraries, different platforms, debug vs release builds — doing this by hand becomes unmanageable. That's what a build system is for.

This project uses a two-layer setup that is standard in the C/C++ world:

```
CMakeLists.txt  (you write this — describes WHAT to build)
      │
      ▼
   CMake         (reads CMakeLists.txt, generates build files)
      │
      ▼
  build.ninja   (generated — describes HOW to build, step by step)
      │
      ▼
   Ninja         (reads build.ninja, runs the actual compiler commands)
      │
      ▼
 renderer.exe
```

You only ever interact with the top (CMake) and bottom (the final binary). The middle is generated automatically.

---

## CMakeLists.txt

This is the only build file you write and maintain. It lives at the project root.

```cmake
cmake_minimum_required(VERSION 3.20)
project(3DRenderer C)

set(CMAKE_C_STANDARD 11)
set(CMAKE_EXPORT_COMPILE_COMMANDS ON)

find_package(SDL3 REQUIRED)

add_executable(renderer src/main.c)
target_link_libraries(renderer PRIVATE SDL3::SDL3)
```

Line by line:

| Line | What it does |
|------|-------------|
| `cmake_minimum_required` | Refuses to run on outdated CMake versions |
| `project(3DRenderer C)` | Names the project; `C` tells CMake this is a C project (not C++) |
| `CMAKE_C_STANDARD 11` | Compiles all C files with `-std=c11` |
| `CMAKE_EXPORT_COMPILE_COMMANDS ON` | Writes `build/compile_commands.json` (used by clangd) |
| `find_package(SDL3 REQUIRED)` | Locates the SDL3 installation on your machine; fails the build if not found |
| `add_executable(renderer src/main.c)` | Defines a build target called `renderer` from `src/main.c` |
| `target_link_libraries(...)` | Links SDL3 into the `renderer` target, also propagating its include paths |

## The Two Build Steps

### Step 1 — Configure

```bash
cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Debug
```

| Part | Meaning |
|------|---------|
| `-B build` | Put all generated files in a `build/` subfolder (keeps the project root clean) |
| `-G Ninja` | Generate files for the Ninja build tool (faster than Make) |
| `-DCMAKE_BUILD_TYPE=Debug` | Set a CMake variable: compile with debug symbols (`-g`), no optimisations |

CMake reads `CMakeLists.txt`, finds SDL3, resolves all paths, and writes the `build/` directory. **You only need to re-run this when `CMakeLists.txt` changes.**

### Step 2 — Build

```bash
cmake --build build
```

CMake hands off to Ninja, which reads `build/build.ninja` and runs the actual compiler commands. **This is what you run every time you change source code.**

## What's in the `build/` Folder

The `build/` folder is entirely generated — you can delete it and recreate it at any time with the two commands above. Never edit files inside it by hand.

```
build/
├── renderer.exe                         ← Your compiled program
├── build.ninja                          ← Ninja's instruction file (don't edit)
├── compile_commands.json                ← Compiler flags for every source file (used by clangd)
├── cmake_install.cmake                  ← Used by `cmake --install` (not needed for learning)
├── CMakeCache.txt                       ← Cached CMake configuration values
└── CMakeFiles/
    ├── renderer.dir/
    │   └── src/
    │       └── main.c.obj               ← Compiled object file for main.c
    └── ...                              ← CMake's internal bookkeeping files
```

### Key files explained

**`renderer.exe`**
The final linked binary. This is what you run.

**`build.ninja`**
Ninja's equivalent of a Makefile. Contains the exact shell commands to compile each `.c` file into a `.obj`, then link the `.obj` files into the final `.exe`. Generated by CMake — do not edit.

**`compile_commands.json`**
A JSON array with one entry per source file, recording the exact compiler command used for that file (flags, include paths, defines). clangd reads this to give IntelliSense accurate results. See [editor-setup.md](editor-setup.md).

**`CMakeCache.txt`**
After the configure step, CMake saves all resolved values here (paths to compilers, SDL3 libraries, etc.) so it doesn't have to search for them again on every build. If something seems stale or broken, deleting the whole `build/` folder and reconfiguring is the standard fix.

**`CMakeFiles/renderer.dir/src/main.c.obj`**
The compiled object file for `main.c`. This is an intermediate binary — it contains the machine code for `main.c` but is not yet a runnable program (it still needs to be *linked* with SDL3 and the C runtime). The linker combines all `.obj` files and libraries into the final `.exe`.

## Debug vs Release

The build type controls compiler optimisation and debug information:

```bash
# Debug — slow binary, full debug info, assertions enabled
cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Debug

# Release — fast binary, optimised, no debug info
cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Release
```

For this project, always use `Debug` while developing. Use `Release` only if you need to benchmark performance.

## Quick Reference

```bash
# First time setup (or after changing CMakeLists.txt)
cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Debug

# Every subsequent build
cmake --build build

# Run the program
./build/renderer.exe

# Start fresh (if something is broken)
rm -rf build/
cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Debug
cmake --build build
```
